本地Spark访问AWS中国区S3的坑和正确姿势

发表于 2016-10-27 | 分类于 数据存储
记录一次尝试使用本地Spark程序读取S3上文件进行处理的踩坑过程，整个调试过程比较折腾，而且事后发现这样做的意义并不大，小文件完全可以从S3上下载到本地，而想用大文件在本机做测试的话，每次从S3拉又不现实，所以建议在实际开发过程当中不要尝试这个方式。

背景信息

公司要在AWS的EMR上跑Spark，而Spark要跑的数据不论怎么来的都是要存在S3上面的，在本地开发Spark代码的时候，每次都从S3上下载数据的话，如果数据很大则几乎不可行，而每次在EMR上开集群做测试又要钱，所以需要在本地Spark上直接访问S3，这也是我踩坑的开始。

这篇文章具有时效性，基于AWS EMR 5.0.0、Hadoop 2.7.2、Spark 2.0.0、Scala 2.7.2，有可能当您看到这篇帖子时坑已经不存在了。

但我个人并不对这个可能抱有多大期望，因为我在进这家公司之前就看到过2015年8月份的一篇帖子，直到大半年后的2016年4月份另一个人也遇到了跟我类似的问题，这个坑迟迟没有修复，而在墙内搜索引擎基本找不到什么有用的信息，墙外网面向的则是AWS国际服，和国服根本不是一个游戏，所以才自己总结了这篇访问S3的正确姿势。

这篇文章用S3表示AWS中国区的存储系统，用小写的s3表示协议，我的测试代码就是从S3读取一个文件做wordCount再写回S3。

在本地Spark代码里使用s3n访问S3

提到Spark这里就要提一嘴Hadoop，Hadoop本身是支持很多种文件访问协议的，在本地Hadoop当中，s3是一种人类不可读的二进制存储格式，AWS已经明确反对使用，s3n和s3a是一种后续退出的人类可读的存储协议，而亚马逊表示s3n已经过时了，所以推荐优先使用s3a方式访问S3存储。

然而我Google到的s3n和s3a怎么都行，有的帖子让你用s3a、有的帖子又说s3n好，那些又根本没说清楚是在本地还是EMR环境运行的，我又怎么知道国服有这个没有意义的大坑？于是就用s3n去尝试访问S3了，悲剧开始了！

在Hadoop当中，为了访问S3文件系统，要导入aws-sdk包，它下有一个叫做s3的子服务供Java语言访问S3，其中会调用hadoop-aws包解析协议，所以也要导入这个包。后来Apache将访问S3的包从hadoop-core包解耦到hadoop-aws包，由AWS维护。

导入hadoop-aws 2.7.2，并导入aws-java-sdk（应该导入1.7.4，导入最新的有坑，后面说），我在尝试的途中还遇到了jackson和joda-time的依赖报错问题，直接去maven装相应版本的库，相对比较好解决，这里不再赘述。

在EMR环境的Spark当中访问S3

这时候我的s3a方式访问S3文件的路径还是写死在代码里的，由于生产当中不可能用web界面提交Spark任务，这条路忽略，只打算尝试以命令行的方式提交job，眼看坑了好久终于解决了问题，高高兴兴地将代码提交到Spark集群，结果还是运行失败，报403错误！

原因是EMR环境上的s3和本地的s3根本不是一个东西！当我们建立EMR集群时，实质上是开了多台EC2机器，里面预装了一套AWS自己的Hadoop生态环境。EMR集群中的Hadoop是通过EMRFS的方式访问S3的，这是个AWS的闭源项目，在它内部会把s3和s3n都翻译成s3，这等同于在本地Hadoop中使用s3n（然而我挂了啊？），而EMR环境是根本不支持s3a的（报403错误），EMR官方文档建议在代码中使用s3来访问文件。

把访问方式改为s3，好了！

另外在EMR环境，无需通过sc.hadoopConfiguration.set设置你的key，因为这已经在EC2机器里设置好了，而且奇妙的是EMR的Spark是可以读到这个配置的，而我本地就不行！>,<

关于放在S3上的jar包

补充一点点并不太坑的东西，关于jar包，EMR上的Spark默认走yarn，故无需跟–master yarn配置，但是默认的–deploy-mode是client方式，这是用来本地测试spark-shell的方式，所以需要手动将这个参数配置为cluster。

如果希望将jar包放在S3上执行并在spark-submit命令中以s3协议指定jar包路径，则必须这样配，否则报错：“Warning: Skip remote jar s3://xxx/xxx.jar.”，这个在官方文档有提到。

直接将jar包上传到集群主节点并运行jar包是没有任何问题的。

一些蛋疼的吐槽

正当我焦头烂额的时候，偶然在CSDN上看到了这个帖子——「spark/hadoop读取s3所需要的外部依赖包」，只给了俩文件，没有帖子内容，也联系不到作者，但是导了包果然好了，而一是在我的Job里导入外部lib不对劲，二是经过比对他给的jar和maven的jar差异很大，三是我不想让问题好得不明不白的，于是一步步去分析依赖，把包拆出去，把依赖转移到maven。

这个过程当中阅读了文章一开始提到的另一篇帖子的后续，笔者也几乎完全记录了整个踩坑过程，和我遇到的情况非常类似，他最后直接改了硬编码的东西重新编译了Spark，在推特上联系他也没找到什么优雅的姿势解决这个坑，重新编译仿佛是个极其痛苦的决定。

经验教训

1.无论如何不要硬编码！
2.共享能解决问题的东西时不要不写说明！
3.Google到的国际服的东西在国服未必好使，反而会把自己绕进去！
